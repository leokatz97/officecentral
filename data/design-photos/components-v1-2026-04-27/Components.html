Buttons — usage spec

Hierarchy & pairing

  • One primary per view. Two primaries side-by-side breaks the red-density rule because both turn red on hover simultaneously when keyboard tabbing.
  • Standard pairing: primary + secondary (CTA + escape hatch). Add tertiary for "or call us" / cancel-style affordances.
  • Order on desktop: primary first, secondary right of it. Reverse on mobile only when the secondary is the more common action (e.g. "Save draft" vs "Publish").

Sizing

  • Default min-height 44px — meets touch-target floor.
  • .bbi-btn--lg 52px for hero CTAs.
  • .bbi-btn--sm 36px for table-row / card-internal actions only — never for primary on a landing screen.

States — required behavior

  • Hover: primary & secondary swap to their *Hover tokens. 120ms ease.
  • Focus: 2px outline in --textColor, 2px offset (driven by :focus-visible in tokens.css). Visible against white AND charcoal — verified in the live strips.
  • Active: same colors as hover + translateY(1px).
  • Disabled: opacity: 0.4; filter: grayscale(1); AND cursor: not-allowed. Never opacity alone — grayscale removes the brand red so disabled isn't mistaken for a hover state.

Don't

  • Don't use red as the resting primary surface — red is a hover/active accent, not a default. (See conflict flag at top.)
  • Don't put two primaries adjacent. Demote one to secondary.
  • Don't use tertiary as the only CTA on a conversion screen — it has no surface, and the brief's quote-request CTA is the conversion pattern.
  • Don't override --buttonRadius. Squared corners are intentional and match the wordmark.
  • Don't apply tertiary's red hover to inverse — fails 2.32:1. Inverse-tertiary hover stays white + underline.

Accessibility checklist

  • Primary default: charcoal/white = 20.10:1 AAA · hover red/white = 4.08:1 AA-large (label is 600 / ≥14px ⇒ qualifies).
  • Secondary default: charcoal/white = 20.10:1 AAA · hover invert = 20.10:1 AAA.
  • Tertiary default: charcoal/white 20.10:1 AAA · hover red-text on white 5.88:1 AA · on inverse stays white 20.10:1.
  • Focus ring 4.93:1+ on every documented surface — meets 1.4.11 non-text-contrast 3:1.
  • Disabled buttons keep aria-disabled="true"; if it's a link-styled tertiary, swap to <button disabled> not a div.

Five field types × four states (default, focus, error, disabled), rendered on both schemes side-by-side. Focus uses --textColor (≥20:1 against canvas) — never --line-color (decorative hairline only). Focus does not change fill, per brief.

Form inputs — usage spec

Sizing & rhythm

  • All text-bearing fields: min-height 44px, padding 10px 14px, font-size 15px. Matches button height for aligned form rows.
  • Textarea min-height 96px (~3 lines), resize: vertical only — no horizontal stretch breaking grid layout.
  • Checkbox & radio box: 18×18, 1px border, 3px radius (check) / round (radio). 10px gap to label.
  • Field stack gap 14px; label-to-control gap 6px; helper-to-control gap 6px.

States — required behavior

  • Default: 1px --inputBorder on --inputBackground. Fully opaque ink.
  • Focus: 2px outline in --textColor at 2px offset, border unchanged. No fill change, no glow. Never --line-color.
  • Error: 1px --error border + 1px inset shadow in --error for optical 2px weight, paired with aria-invalid="true" and a .bbi-helper--error message. Border and helper are independent channels — color-blind users see the icon + bold weight.
  • Disabled: opacity: 0.4; filter: grayscale(1); cursor: not-allowed. Same rule as buttons; never opacity alone.

Inverse-scheme error rule (flagged)

  • On charcoal, --error body text fails 4.5:1. Resolution: keep red on the border (non-text, 3:1 passes) + red icon glyph; flip helper-prose to --textColor. Multi-channel error remains, no token invented.
  • If a future spec wants red prose on inverse, that requires a new --errorOnInverse token (chat 1).

Required-field convention

  • Asterisk in .req (red, 2px right margin). Never the only signal — pair with explicit "Required" in the helper or a server-side validation message. Never use red label text alone (red headings are banned).

Don't

  • Don't use --line-color for focus — it's 1.22:1 hairline only. Use --textColor.
  • Don't change input fill on focus or error — borders + helpers carry the signal.
  • Don't put the asterisk inside the helper. It belongs in the label.
  • Don't use the native browser select chevron — it's beige-tinted in some skins. Use the CSS-drawn caret in .bbi-select.
  • Don't override --inputRadius. 4px squared corners match buttons + brand wordmark.

Accessibility checklist

  • Every input has a programmatic label (<label> wrap or aria-labelledby).
  • Errors set aria-invalid="true" and link to helper via aria-describedby.
  • Focus ring 20.10:1 on both schemes — far above 1.4.11's 3:1.
  • Hidden native checkbox/radio retains keyboard access; the visible .box is decorative.
  • Disabled controls keep readable contrast (grayscale preserves it where opacity alone would not).

Cards — usage spec

Surface & lift

  • All cards: --cardBackground, 1px --borderColor, --cardRadius: 8px, no shadow at rest.
  • Hover: --shadow-sm (≤8% opacity per token spec). On inverse the shadow barely registers — border + plate carry separation. This is intentional, not a bug.
  • Card-as-link: pair with :focus-visible ring (handled globally in tokens.css). Whole card is clickable; embed CTA as visual affordance only.

Product card — hard rules

  • 16:9 media at the TOP. No exceptions. Use aspect-ratio: 16 / 9 so cards align perfectly in a grid even with mixed image dimensions.
  • Sale badge → top-LEFT. Sold badge → top-RIGHT. Different corner = different meaning. Never both in the same corner.
  • Sold-out: filter: grayscale(1) + opacity: 0.7 on media, line-through on price, CTA copy swaps to "Request a quote" (the unbuyable-item pattern; PDP variant lives in chat 07).
  • Image must have meaningful alt; the placeholder pattern is for missing assets only and ships a striped texture, never a solid block.

Feature card — hard rules

  • Numbered (01, 02, 03…) in mono, colored --saleBadgeBackground. This is the ONLY place red text is allowed in body copy — it mirrors the baseline homepage. Do not propagate this to other contexts.
  • No media. Title 22px (--fs-h3 desktop), denser padding (--space-8).
  • Typically renders in a 2- or 3-column row sharing internal borders. Standalone use is allowed but rare.

Don't

  • Don't add shadow at rest. Hover only. Never elevate above --shadow-md.
  • Don't put the price above the title. Title is the primary scannable element.
  • Don't put a sale badge AND a sold badge on the same product — sold supersedes everything else.
  • Don't render product cards without media. If imagery isn't ready, ship the striped placeholder, not a solid color block.
  • Don't override --cardRadius. 8px matches the brand wordmark + image tile system.
  • Don't use --saleBadgeBackground as red text outside the feature-card numeric label and the badge component proper.

Accessibility checklist

  • Card-as-link uses <a> wrap or an ::after click target — keyboard accessible.
  • Badge stubs render bold ≥11px white-on-color: sale red 4.83:1 AA-large, sold gray 6.74:1 AA. Verified in tokens v1.
  • Sold-out grayscale + line-through is multi-channel — color-blind users still see the line-through and the changed CTA copy.
  • Card focus ring uses --textColor, 2px @ 2px offset; passes 1.4.11 on both schemes.

04 — Badges

tokens v1 · locked

Five badge variants — sale, new, preorder, sold, custom — rendered standalone and on a sample product card showing corner-placement rules. Sale top-LEFT, sold top-RIGHT (separate corner = different meaning, never overlap).

No brightness auto-flip. Each badge has a deterministic, token-defined label color (per comments in tokens v1). An algorithmic luminance flip would risk pushing the sale badge's red/white pairing (4.08:1, AA-large) below threshold or accidentally inverting against a token shift. Pairings are hard-coded.
Inverse-scheme flag (flagged, not silently fixed). preorder (#FFFFFF) and custom (#FAFAFA) are white-filled badges. On scheme-inverse they read as primary stickers/flags — visually louder than intended. Mitigation: 1px white outline on inverse to integrate them into the dark canvas without inventing a new token. If badges-on-inverse become a frequent pattern, recommend a chat-1 token revision adding --preorderBadgeBackground-onInverse / --customBadgeBackground-onInverse.

Badges — usage spec

Placement (hard rule)

  • Sale → top-LEFT. Sold → top-RIGHT. Different corner = different meaning. Never the same corner, never swapped, never centered.
  • new, preorder, custom stack with sale on the left under the sale badge — they're attribute badges, sale is a state badge.
  • Sold supersedes all other badges. A sold-out card never shows sale/new/preorder/custom.

Stacking

  • Maximum two badges per corner. Three is visual noise.
  • Within left stack: sale always on top, attribute badges (new/preorder/custom) below.
  • Inset from media corner: --space-3 (12px) on all sides.

Type

  • 11px / 700 / uppercase / +0.08em letter-spacing. Never lowercase, never bold-italic.
  • Padding 5px 8px, border-radius 2px (sharper than buttons — reads as a sticker, not a chip).
  • Min copy length 3 chars, max ~14 chars. Long copy belongs in helper text, not a badge.

Color flip — explicit, not algorithmic

  • No color_brightness auto-flip. Pairings are hard-coded per token comments. An algorithmic flip would push sale 4.08:1 across threshold under any future hue tweak.
  • If a new badge variant is needed, the bg + label color must be defined in tokens together — never just the bg.

Inverse-scheme caveat

  • preorder & custom (white-fill) read loud against charcoal — they're high-contrast like a primary CTA. Mitigated here with a 1px white border that integrates them into the canvas.
  • If badges-on-inverse becomes a recurring pattern (e.g. inverse-hero product cards), recommend chat-1 revision adding inverse-specific badge tokens.

Don't

  • Don't use the badge component as a navigation tag, a category chip, or a status pill. It's reserved for product-state communication.
  • Don't override the label colors. Each pairing is contrast-verified.
  • Don't put a sale badge on a sold-out card. Sold supersedes.
  • Don't render in body copy line-height — badges sit on imagery, not in prose.
  • Don't auto-flip with color_brightness. See note above.

Accessibility checklist

  • Badges are visual labels, not interactive. If they convey unique info (e.g. sold), pair with text in the card body — color/badge alone is never the only signal.
  • Sold-out cards combine: sold badge (right) + grayscale media + line-through price + CTA copy "Request a quote" — four redundant channels.
  • Min font 11px is the floor for AA-large at 700 weight; never go smaller.
  • Sale badge contrast (4.08:1) is AA-large only — keep it bold uppercase ≥11px, and never use sale color for body copy.

05 — Header

Desktop and mobile (375px) headers, both schemes. Megamenu is single-column with the 9 categories from the brief. Phone number renders as charcoal click-to-call. Request a quote is the only red CTA — and the cart count dot is the only other red element.

Nav structure (5 items)

  • Order: Shop Furniture · Industries · Brands · Services · About.
  • Shop Furniture → single-column megamenu, 9 items, header "Business Furniture" → /collections/business-furniture.
  • Industries / Brands / Services / About → simple dropdowns.
  • Archived 2026-04-25: Healthcare / Education / Government / hospitality vertical landing pages. Do not surface in the megamenu.

Color discipline

  • Request a quote = the ONLY red CTA in the header.
  • Cart count dot = the only other red element. Never both absent and never duplicated.
  • Phone is charcoal text (tel: link). Hover shifts to --headerHoverColor red — same hover treatment as nav links.
  • No red on default nav. Red appears only on hover/focus.

Mobile (375px)

  • Hamburger right, logo left. 60px min-height (vs 72 desktop).
  • Vertical accordion. Same 5-item structure; same 9 megamenu items under Shop.
  • Drawer surface = --submenuBg; expanded panel fill = --alternateBackground; bottom-pinned phone + CTA so they're reachable from a thumb.
  • Only one item open at a time (single-disclosure pattern).

Don't

  • Don't put more than one red CTA in the header. Demote secondary CTAs to charcoal text or move them out.
  • Don't recolor the logo. Don't render on a red plate.
  • Don't expand the megamenu to 4 columns of verticals — they're archived.
  • Don't render the header on scheme-inverse in production. Tokens fail on charcoal.
  • Don't lower phone contrast — it must read as a click-to-call link, not a styling artifact.

Accessibility checklist

  • Phone is <a href="tel:…">. Always call-actionable.
  • Megamenu/dropdown buttons use aria-expanded + aria-controls.

  • Tab order: logo → nav items → utility (phone, cart, CTA). Skip-to-content link should precede.

07 — Quote-request CTA

tokens v1 · locked

The quote-request CTA exists in two forms. The standalone section is a marketing band that anchors the bottom of category pages, the homepage, and editorial layouts. The inline PDP variant replaces add-to-cart entirely on unbuyable items — most of the catalog, since BBI sells through quote, not transaction.

Why two variants. The standalone band closes a story (after browsing); the PDP block opens one (after intent on a specific SKU). Same component family, different hierarchy: standalone leads with a heading, PDP leads with the action. The button styling is the only piece that's literally larger on PDP — everything else is contextual copy and trust signals.
OECM trust line — flagged. "OECM-vendor of record" is included as the lead trust signal because BBI is on the OECM (Ontario Education Collaborative Marketplace) roster — a meaningful credential for the Education and Government industries. If this is not currently true, swap with another verifiable signal: years in business (since 1962), client count, or contract reference. Don't ship unverifiable claims.
Red discipline maintained. CTA button is the only red element in the standalone section. In the PDP variant, the small "Why request a quote?" eyebrow + the response-time icon + the phone-fallback hover state share the same red — but the button still anchors. Cart icon is absent from PDP because the item isn't add-to-cart-able; that's the entire reason this variant exists.

Quote-request CTA — usage spec

When to use which

  • (a) Standalone section — bottom of category pages, homepage, editorial layouts, and as a closing band on long-scroll content. Always full-bleed; never side-by-side with another CTA.
  • (b) Inline PDP variant — replaces add-to-cart on every product where pricing depends on configuration, quantity, freight, or contract — i.e., most of the BBI catalog. Sits in the PDP's right rail, where add-to-cart would.

Hierarchy

  • Standalone: heading-led. Visitor scans heading → button. Trust lines support after-the-click commitment.
  • PDP: action-led. Visitor lands on a specific SKU with intent — the "why request a quote?" microcopy is the answer to a question they're already asking. Button is full-width, 56px tall, and visually the second-loudest element after the product image itself.

Sizing

  • Standalone heading: clamp(32px, 4.2vw, 56px) · 700 · -0.01em.
  • Standalone subhead: 17px · 1.5 · max-width 52ch.
  • Standalone CTA: .bbi-btn--lg (52px min, 16px font).
  • PDP heading: 20px · 700.
  • PDP CTA: 56px tall · full-width · 16px font · centered label. Larger than the standard --lg button because it carries the entire conversion role.
  • PDP card max-width: 460px. Above that, the right rail itself is the constraint.

Copy rules

  • Standalone heading: outcome-framed, ≤8 words. "Get a quote, not a cart total." sets the model.
  • PDP eyebrow: always "Why request a quote?" — answers the visitor's implicit question.
  • PDP heading: one sentence, ≤14 words, names the specific reason ("configurable", "freight-dependent", "quantity-priced").
  • PDP microcopy: "We respond within 1 business day" is mandatory. If you can't honor this SLA, change the copy — don't drop it.
  • Button label: "Request a quote" on standalone (sentence case); "Request a Quote" on PDP (title case, action-emphasis). The casing is intentional — PDP buttons across e-com conventionally use title case, and visitors scan for it.

States (button)

  • Inherits all five states from .bbi-btn--primary (default · hover · focus · active · disabled). No new state work.
  • On the standalone (charcoal), the focus ring uses --textColor = white. On the PDP card (white), it uses --textColor = charcoal. Both already covered by the button component.

Don't

  • Don't pair the standalone CTA with another red element in the same band. The button is the only red.
  • Don't use the PDP variant on items that are genuinely add-to-cart-able. If a SKU has a fixed price + ships standalone, give it a real cart button.
  • Don't shrink the PDP button below 56px. Its size is part of the spec.
  • Don't replace "OECM vendor of record" with a generic "trusted by businesses" line. Trust signals are specific or they're noise.
  • Don't add a secondary button alongside the primary CTA. Phone fallback is a text link, not a button.
  • Don't use the standalone section as a header — it's a closer. Header is its own component (05).

Accessibility

  • Standalone: white on charcoal = 20.10:1 AAA. White-at-78%-opacity on charcoal = ~13.9:1 AAA.
  • PDP red eyebrow on white: #D4252A = 4.93:1 AA-large; eyebrow is 12px / 600 — borderline. Mitigation: 600 weight + 0.08em tracking lifts perceived weight above the AA-large threshold. Flagging for the audit.
  • Phone numbers use tel: schemes; "1 business day" microcopy is real text, not an image.
  • Tab order on PDP: button → phone link → project-list link. The eyebrow + heading are decorative within the card and skipped by tab.
  • Standalone heading is an <h2>. PDP heading is an <h3> because it sits inside the PDP's <h1> + <h2> outline.
